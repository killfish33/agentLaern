## 2. 项目背景
当前已有基础 Agent Demo，但缺乏可用于 ToB 场景的全栈工程能力。为支撑目标岗位要求，需要从“可对话”升级到“可稳定、可扩展、可观测、可交付、可上线”的 Agent 全栈系统（Next.js）。

核心问题：
1. 流式链路可用但不稳定（断流、重试、幂等能力不足）
2. Tool Calling 缺少可视化与容错编排
3. 动态 UI 渲染缺少 Schema 约束与安全机制
4. ToB 权限、性能、可审计能力不足
5. 缺少完整指标体系与故障复盘能力

---

## 3. 项目目标

### 3.1 业务目标
- 交付一个可演示且可上线的 Agent 全栈项目，覆盖“对话-工具-动态 UI-权限-监控-部署”闭环。
- 输出可用于面试/汇报的证据材料（架构图、指标、复盘、演示视频）。

### 3.2 产品目标
- 用户可以通过流式对话完成任务，并看到工具调用状态与结构化结果。
- 系统在网络波动与异常场景下可恢复并保障一致性。
- 动态卡片/表单可由 Schema 驱动渲染，具备安全校验。

### 3.3 技术目标（量化）
- 首 Token 时间（TTFT）：
  - P50 ≤ 1.2s，P90 ≤ 2.5s
- 流式会话成功率：≥ 99%
- 重连恢复成功率：≥ 95%
- 重复消息渲染率：≤ 0.1%
- Tool 调用可视化覆盖率：100%

## 6. 核心使用场景
1. 用户发起问题，系统流式返回结果并可中途取消/重试
2. Agent 调用工具（查询、分析、生成），前端展示工具状态与结果
3. 返回 Schema，前端动态渲染卡片/表单并支持继续交互
4. 不同角色查看同一数据时，字段可见范围不同
5. 异常发生时，系统记录 trace 并可快速定位

## 7. 功能需求（详细）

## 7.1 流式对话模块
### 功能点
- 发送消息、接收流式 chunk、消息合并渲染
- 支持中断、重试、清空上下文
- 连接状态展示（连接中/稳定/重连中/失败）

### 规则
- 每条消息必须有 `messageId`，流式事件有 `seq`
- 按 `messageId + seq` 去重
- 状态流转仅允许：
  - `pending -> streaming -> done`
  - `pending/streaming -> error`

### 异常处理
- 网络断开：自动重连（指数退避）
- 超时：提示并支持手动重试
- 非法事件：丢弃并上报日志

---

## 7.2 会话与上下文管理
### 功能点
- 会话列表、当前会话切换、历史记录加载
- 上下文窗口控制（长度/Token 近似阈值）
- 历史摘要压缩（保留关键信息）

### 规则
- 会话隔离：不同会话上下文不得互串
- 历史裁剪需保留系统消息与最近关键轮次

---

## 7.3 Tool Calling 可视化
### 功能点
- 展示工具名、参数、执行状态、耗时、结果摘要
- 工具调用失败可重试
- 工具结果可回填到消息流中

### 状态
- `queued` / `running` / `success` / `failed` / `timeout`

### 交互要求
- 用户可展开查看参数与返回
- 失败态需给出可理解错误信息

---

## 7.4 Generative UI（Schema 驱动）
### 功能点
- 根据 Schema 渲染组件（卡片、表单、图表容器）
- 组件参数校验（类型、范围、必填）
- 支持事件回传（提交、点击、筛选）

### 安全要求
- 严格组件白名单
- 禁止任意脚本注入
- 富文本做 XSS 过滤

---

## 7.5 插件系统与 JS-SDK
### 功能点
- 插件注册/挂载/卸载
- SDK 提供消息通信、事件总线、埋点上报
- 支持在微前端容器中作为子模块运行

### 兼容性
- 主应用与插件版本协商（最小兼容版本）

---

## 7.6 ToB 权限与数据展示
### 功能点
- RBAC 角色权限控制
- ABAC 条件策略示例（按部门/数据域）
- 字段级权限（敏感字段掩码/隐藏）
- 大表格性能优化（虚拟滚动、分页）

---

## 7.7 可观测性
### 指标
- TTFT、消息完成时延、工具成功率、错误率、重连次数

### 日志
- 统一字段：`traceId`、`sessionId`、`messageId`、`toolCallId`

### 可视化
- 基础监控面板：趋势图 + 异常统计

## 8. 非功能需求

## 8.1 性能
- 首屏可交互时间（TTI）≤ 3s（开发环境可放宽）
- 长会话（100+ 轮）保持可用，无明显卡顿

## 8.2 稳定性
- 单次异常不导致全局崩溃
- 关键路径均有兜底 UI（失败提示、重试入口）

## 8.3 安全
- 输入输出统一校验
- 前端不信任工具返回，必须二次校验

## 8.4 可维护性
- 模块边界清晰：`chat` / `tool` / `schema-ui` / `plugin` / `monitor`
- 关键模块有最小测试覆盖（状态机、去重、渲染映射）

## 10. 核心数据模型（前端视角）

```ts
type MessageStatus = 'pending' | 'streaming' | 'done' | 'error'

type StreamEvent = {
  messageId: string
  seq: number
  type: 'token' | 'tool_call' | 'tool_result' | 'done' | 'error'
  payload: unknown
  ts: number
}

type ChatMessage = {
  id: string
  role: 'user' | 'assistant' | 'system'
  content: string
  status: MessageStatus
  traceId?: string
}

type ToolCallState = {
  toolCallId: string
  toolName: string
  status: 'queued' | 'running' | 'success' | 'failed' | 'timeout'
  durationMs?: number
  errorMsg?: string
}
```

---

## 11. 关键流程（简述）
1. 用户发送消息 -> 创建 `pending` 消息
2. 建立流连接 -> 接收 chunk -> 更新为 `streaming`
3. 若触发工具调用 -> 进入 Tool 状态链路
4. 工具结果回填 -> 继续流式输出
5. 收到结束事件 -> 消息变为 `done`
6. 异常发生 -> 消息置 `error`，展示重试入口

## 14. 风险与应对
1. 流式协议不稳定
   - 应对：先 SSE 后 WS；统一事件适配层
2. 工具返回结构不一致
   - 应对：Schema 校验 + fallback 渲染
3. 动态 UI 安全风险
   - 应对：白名单 + 参数校验 + XSS 过滤
4. 周期紧张
   - 应对：MVP 优先，增强功能按优先级递进
